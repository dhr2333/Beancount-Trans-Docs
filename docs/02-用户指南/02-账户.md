账户像一张地图的骨架。每当账单被解析成 Beancount 语句，平台都会问一句：" 这一笔该放到哪个坐标？" 答案，就是账户。理解账户并不需要会计背景，它更像是整理收纳：把散落在桌面的东西放回盒子，家里自然井井有条。

## 一、账户在 Beancount-Trans 中扮演的角色

- 账户定义了资金的去向和来源，帮助系统在报表里区分 " 花在哪里 "" 赚了多少 "" 还欠什么 "。
- 平台在注册时会为每位用户铺好一套官方账户模板，就像开箱即用的衣柜，常见分类都已经准备好。
- 当映射规则匹配到一条交易时，它会把交易轻轻放进对应的账户，于是 " 账单 → 分析 → 报表 " 这条流水线才算闭环。

## 二、树状结构：把账户当成一棵成长的树

```mermaid
graph TD
    A[根账户] --> B[Assets 资产]
    A --> C[Liabilities 负债] 
    A --> D[Equity 权益]
    A --> E[Income 收入]
    A --> F[Expenses 支出]
    
    B --> B1[银行卡]
    B --> B2[余额宝]
    B --> B3[微信零钱]
    
    F --> F1[Food 餐饮]
    F1 --> F11[早餐]
    F1 --> F12[午餐]
    F1 --> F13[晚餐]
    
    F --> T1[TransPort 交通]
    T1 --> T11[私家车]
    T1 --> T12[公共]
```

想象一棵 " 财务树 "——五个根分出粗壮的树干，树干再分枝成细致的科目。

- 冒号区隔层级，`Expenses:Food:Breakfast` 表示 " 支出 → 餐饮 → 早餐 "。
- 如果枝条不存在，平台会自动为您补上，省去手动搭骨架的麻烦。
- 改动上一层名称时，树枝会跟着调整，保持路径的完整性。

## 三、账户的一天：从创建到归档

1. **诞生**：当您添加一个全新路径时，系统会检查父级是否存在并自动补齐。官方模板本身就覆盖了大多数日常场景，只需在需要时再延伸。
2. **活跃**：启用状态意味着它正在接收交易。若暂时不想使用，切换为停用即可，相关映射也会休眠。
3. **归档**：如果决定告别某个账户，可以先把映射迁移到新的去处，再将它关闭或删除，账本结构依旧保持连贯。

## 四、和映射、解析、报表的协同

- **🎯 映射**：关键字匹配后投递到目标账户
- **🔍 解析**：参照现有账户整理新鲜数据
- **📊 报表**：Fava 根据账户结构生成各种图表

## 五、设计属于自己的账户小技巧

> [!NOTE] 层次分明原则
> 保持2-3层结构即可，只有当现有账户无法回答"这笔钱为何而来/用往何处"时，再新增子账户。


- 从 " 粗分类 " 起步，再逐步细分。先有 `Expenses:Food`，再视需要拆出 `Breakfast`、`Dinner`。
- 给经常往返的资金留出专属路径，例如网银提现、钱包充值等，后续统计更直观。
- 账户名称尽量语义清楚、全部英文或全部中文一致，方便与自动化工具协同。

## 六、常见问题 (FAQ)

**Q1: 账户太多会不会乱？**

**A:** 记账的关键是层次分明。建议保持两到三层结构即可，只有当现有账户无法回答 " 这笔钱为何而来 / 用往何处 " 时，再新增子账户。

**Q2: 想迁移旧交易到新的账户应该怎么做？**

**A:** 可以先调整映射规则，再重新解析原账单；如果是少量记录，也可以直接在账本中手动替换账户字段。两种方式都能让交易与最新的账户体系保持一致。

**Q3: 账户和标签、备注如何搭配使用？**

**A:** 账户负责 " 把交易放进抽屉 "，标签与备注则提供更多语义，例如标记商务消费或报销事项。两者结合后，Fava 的过滤器就能回答更多维度的问题。
